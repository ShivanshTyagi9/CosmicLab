Fused AIS Decoder Architecture

The AIS Decoder pipeline is designed to be a highly resilient, fused-system architecture that seamlessly processes both NMEA/NMTR and DGLL payloads concurrently. It leverages a Master/Worker model to prevent bottlenecks and deadlocks while maximizing core utilization.

Pipeline Components & Optimizations

  • Mounted RAM (tmpfs): The Raw/ ingress directory is mapped to a high-speed RAM disk, drastically reducing I/O latency when processing incoming .txt payloads.
  • Watchdog Observer: Constantly monitors the Raw/ ingress directories for new or modified payloads.
  • File Classifier: Scans incoming filenames to determine if they are NMEA/NMTR formatted or DGLL formatted, subsequently routing them to the correct pending queue.
  • Pending Queues: Two thread-safe queues that securely hold the backlog of work for the respective worker pools.
  • CPU Core Pinning: Each worker process is assigned to a dedicated, physically isolated CPU core to ensure zero thread contention and maximize decoding throughput.
  • Worker Pools: Dedicated parallel processes running decoder.py and dec_DGLL.py. These workers securely pull payloads from the queues and parse the raw string sentences into structured data dictionaries.
  • Result Queue: A shared thread-safe queue where the parsing results (or error traces) are passed back to the Master Orchestrator for tracking and cleanup.
  • Processed Output: The final decoded records are appended into clean, partitioned dynamic_ais.txt and static_ais.txt CSV repositories for downstream command core analytics.
