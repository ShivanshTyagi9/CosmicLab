EchoDB: Persistent, Temporal Memory Layer for LLMs

EchoDB

Persistent, Temporal Memory Layer for LLM-Powered Conversational Agents

Without memory, every conversation with an AI starts from zero. EchoDB changes this by acting as a persistent, temporal memory layer. Drop it into any chat loop, and your assistant will remember facts, track how they change over time, extract structured knowledge from conversations, and retrieve the right context at the right moment.

Runs fully locally with llama.cpp (no API keys required), or connects to OpenAI / Anthropic with a simple configuration change.

Key Capabilities

  • Structured Recall — The agent knows not just that the user mentioned something, but what kind of thing it was: a fact, a preference, a task, a technical procedure, or an episodic event.
  • Temporal Awareness — When facts change (e.g., “I moved from Seattle to New York”), old facts are preserved with timestamps, and a new version is created. The agent retrieves the latest version but can trace the full timeline.
  • Automatic Entity Linking — Pronouns and references resolve to the same person (e.g., “Alex”, “he”, and “the user”). Related concepts are wired as graph edges and retrieved together.
  • Decay & Reinforcement — Episodic memories fade if never recalled again, while stable facts and procedural workflows persist indefinitely. Accessing a memory counteracts its decay.
  • Portable Snapshots — Save the entire memory state (graph structure + embeddings + KB sections) into a single .umc file and restore it anywhere instantly.
  • Obsidian Integration — Write human-readable Markdown files representing entities and relationships directly to a local Vault folder.

Architecture: The Best of Both Worlds

EchoDB merges two distinct paradigms of memory architectures:

  1. Vault Pipeline — Handles rich extraction. It scans a conversation and pulls out entities, relationships (works_at, member_of), code artifacts, and multi-step procedures.
  2. Capsule Engine — Handles a sophisticated temporal graph. It tracks decay rates, manages version history via update chains, and retrieves items using a hybrid scoring algorithm.

Under the Hood: Deep Dive

1. Identity Normalization

The LLM often generates a generic "User" entity. EchoDB resolves first-person pronouns ("User", "me", "I") into the speaker’s true name (e.g., "Priya"). This mapping is cached in a persistent entity registry, ensuring facts remain coherent across turns.

2. Technology De-fragmentation

To prevent the graph from exploding with empty technology stubs (like creating a file for Kafka or FastAPI with no independent facts), EchoDB collapses stubs. If a technology has no outgoing relations and is just a relation target, it gets absorbed directly into its parent entity’s facts (e.g., Fraud Detection API gains the fact "uses Kafka (event stream)").

3. Temporal Versioning (The UPDATES Chain)

When a new fact contradicts an existing fact, EchoDB classifies this relationship and builds a temporal version chain:

  • The old node is marked is_latest = False and updated with a superseded_at timestamp.
  • A TEMPORAL_CHAIN edge is created pointing from the new node to the old node.
  • You can query the entire history of a fact using get_full_timeline(node_id) or visualize it directly.

Memory Strengths and Decay

EchoDB models memory decay using an exponential decay formula combined with retrieval reinforcement:

strength=min⁡(1.0, (salience×e−decay_rate×elapsed_seconds+min⁡(0.25, 0.025×access_count))×confidence)\text{strength} = \min\left(1.0, \, \left(\text{salience} \times e^{-\text{decay\_rate} \times \text{elapsed\_seconds}} + \min(0.25, \, 0.025 \times \text{access\_count})\right) \times \text{confidence}\right)strength=min(1.0,(salience×e−decay_rate×elapsed_seconds+min(0.25,0.025×access_count))×confidence)

Each memory type features distinct decay behavior:
